# Local Wikidata dataset (offline answers for the example)

A second Virtuoso instance that holds just the Wikidata data needed to answer the
example question — so the example works **offline and deterministically**, without
depending on the public Wikidata Query Service (WDQS), which aggressively
rate-limits (HTTP 429) repeated queries.

## How it works

- **`wikidata-dataset`** (docker-compose) — a `wseresearch/qanary-virtuoso`
  instance on dedicated ports **1112** (ISQL) / **8891** (HTTP/SPARQL) so it
  coexists with the Qanary triplestore Virtuoso (1111/8890). It is configured
  (`.env`) with `VIRT_SPARQL_DefaultGraph=urn:wikidata`, so SPARQL queries
  **without** an explicit `FROM` (as generated by `QB-Wikidata`) resolve against
  the graph that holds the data.
- **`wikidata-dataset-loader`** — a one-shot container that waits for the dataset
  Virtuoso and then loads `wikidata-subset.ttl` into `<urn:wikidata>` via the
  SPARQL Graph Store HTTP Protocol (`PUT`, so it is idempotent across restarts).
- **`QE-SparqlExecuter`** is pointed at `http://localhost:8891/sparql`
  (its `.env`). Set it back to
  `https://query.wikidata.org/bigdata/namespace/wdq/sparql` to use public WDQS.

## The data

`wikidata-subset.ttl` is **real Wikidata data** copied via the Wikidata Action
API (not WDQS). It is regenerated with:

```bash
cd Wikidata-dataset && python3 build-dataset.py
```

`build-dataset.py` fetches, for each subject entity (currently `Q590076`,
"Hawaiian pizza"): its label, its entity-valued truthy statements, the labels of
the referenced objects, and each used property's `wikibase:directClaim` + label —
exactly what the `QB-Wikidata` answer query joins on. Add more subject QIDs to
`SUBJECTS` in the script to cover further test questions.
